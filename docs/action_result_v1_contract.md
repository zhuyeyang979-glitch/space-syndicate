# ActionResult v1 public contract

ActionResult v1 is a presentation contract for two live UI actions: `card_group_ready` in the `card_resolution` family and `district_card_purchase` in the `card_market` family. Card Resolution remains the readiness owner. The v0.6 CardFlow transaction, player-state port, market quote authority, and inventory policy remain the sole purchase owners. `ActionResultPresentationService` only maps a validated public receipt or failure code to fixed public copy.

Every public result contains `success`, `failure_code`, `title`, `explanation`, `consequence`, `suggested_action`, `focus_target`, `relevant_cost`, `relevant_requirement`, and `affected_entity_ids`, plus the fixed v1 action identity and status. The first adopter recognizes `group_ready_committed`, `player_unavailable`, `queued_entry_missing`, `group_window_closed`, `already_ready`, and `ready_rejected`. The second adopter recognizes an exact anonymous-purchase receipt and the public failures `purchase_market_unavailable`, `purchase_listing_changed`, `purchase_source_unavailable`, `purchase_terms_unavailable`, `purchase_funds_unavailable`, `purchase_inventory_unavailable`, and `purchase_conflict`.

The purchase success input is exactly `{event_code: anonymous_purchase_committed, district_index: <non-negative integer>, price_cash: <non-negative integer>}`. The locked listing price is public and lets the buyer understand the committed cost. Card identity, buyer identity, cash balance, inventory/hand data, owner transaction id, quote id/fingerprint, and AI weights are never accepted by the presentation request. Owner failure reasons are collapsed to the stable public failure codes above and are never echoed verbatim.

The contract rejects unknown fields, malformed action identity, Objects/Callables, and recursively private keys or sentinel values. It never publishes a player index, exact cash, hand/discard/slot state, owner truth, AI plan/score/weight, authorization payload, private quote binding, or private receipt. A ready result may identify only `resolution:<non-negative integer>`; a committed purchase may identify only its public `district:<non-negative integer>` source.

The retired `bid_set_*` buttons were a stale UI path after priority-bid authority had already been removed from the queue owner. This cutover does not recreate bidding rules or escrow; it removes Main's dead dispatch/helpers and adopts the real group-ready transaction. Restoring a true priority bid would require a separate authoritative owner and rule acceptance phase.

For regional facility purchases, Main no longer owns a facility-purchase helper, price/cash success sentence, or generic failure sentence. It dispatches the selected public card id to the Coordinator action, receives only ActionResult v1, presents it, and triggers the existing scenario signal on success. The Coordinator first lets CardFlow finish the atomic transaction, then projects either the anonymous public receipt or the owner failure code; presentation never gates or mutates the purchase.
